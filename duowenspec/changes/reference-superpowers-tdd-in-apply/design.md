## Context

当前 apply 指令由 schema 的 `apply.instruction` 与已有制品上下文拼接生成，能指导实现但缺少“执行节奏”约束。团队希望在不破坏默认行为的前提下，引入可选的 TDD 模式，并将该模式稳定体现在 `dwsp instructions apply` 输出中。

## Goals / Non-Goals

**Goals:**
- 在 apply 阶段支持可配置的开发模式。
- 提供内置 `superpowers-tdd` 模式，明确先测后码后验流程。
- 让 apply 文本与 JSON 输出都能表达当前模式，便于人和自动化消费。
- 未配置时保持当前行为完全兼容。

**Non-Goals:**
- 不改变任务勾选与状态计算逻辑。
- 不新增独立的 tdd schema 或重做现有 schema 依赖图。
- 不改动 archive/verify 的执行语义。

## Decisions

1. 决策：将“开发模式”作为 apply 指令层配置，而不是新增 schema。
- 理由：用户诉求集中在 apply 节点执行方式，局部扩展成本低，兼容性最好。
- 备选方案：新增独立 tdd schema。
- 未选原因：会引入更多迁移成本，并放大对现有流程的改造范围。

2. 决策：内置命名模式 `superpowers-tdd`，并输出明确三段式步骤（先失败测试、再最小实现、再回归验证）。
- 理由：可直接满足“引用 superpowers 中的 tdd 模式”诉求，减少团队自定义分歧。
- 备选方案：只允许自由文本模式。
- 未选原因：自由文本难保证一致性，测试断言也更脆弱。

3. 决策：apply JSON 输出新增 `developmentMode` 字段，并在 `instruction` 中体现模式化指引。
- 理由：兼顾机器可读与人可读；下游工具可以稳定识别当前模式。
- 备选方案：仅在 instruction 文本中追加模式描述。
- 未选原因：纯文本不利于自动验证与兼容处理。

## Risks / Trade-offs

- [风险] 模式文案变化导致依赖固定字符串的脚本失效 → 缓解：新增结构化字段并在文档说明兼容策略。
- [风险] 团队误以为启用模式会自动生成测试代码 → 缓解：在模式说明中明确“只提供执行指引，不自动产出代码”。
- [权衡] 增加一层配置会提高理解门槛 → 缓解：默认关闭、开箱不变，仅在用户显式启用时生效。
